Você é um assistente especializado em segurança de software e IA. Preciso da sua ajuda para desenvolver um projeto acadêmico da FIAP (Pós Tech - Software Security, Fase 5 - Hackaton).

## Objetivo do Projeto

Desenvolver um MVP que recebe uma imagem PNG de um diagrama de arquitetura de software (AWS, Azure ou GCP) e gera automaticamente um Relatório de Modelagem de Ameaças baseado na metodologia STRIDE.

## Pipeline (4 passos, 7 sub-etapas)

O sistema deve executar os seguintes passos em sequência:

**Passo 1 — Identificar provedor cloud:**
- Receber a imagem e identificar se é AWS, Azure ou GCP
- Notebooks 01-03: Claude Vision API (multimodal) | Notebook 04: Qwen2.5-VL-72B (local)
- Output: string "AWS", "Azure" ou "GCP"

**Passo 2 — Detectar componentes (3 sub-etapas):**

*Passo 2a — Detecção visual com Florence-2:*
- Usar Florence-2 (microsoft/Florence-2-large-ft, fine-tuned, 770M parâmetros) com múltiplas estratégias de detecção para diagramas técnicos:
  - OCR_WITH_REGION: detecta textos (nomes dos serviços) com bounding boxes
  - CAPTION_TO_PHRASE_GROUNDING: localiza termos específicos de componentes cloud (1 por query)
  - REGION_PROPOSAL: propostas genéricas de regiões
  - DENSE_REGION_CAPTION: regiões com legenda densa
  - OD: detecção de objetos (complementar)
- Tiling 3x3 com 20% overlap para capturar componentes menores
- Filtros para diagramas (min_area_ratio=0.0005, max_area_ratio=0.03, min_size=5px)
- NMS (IoU > 0.5) para remover duplicatas geométricas

*Passo 2b — Classificação:*
- Recortar cada região detectada (crop) e classificar com nome exato do serviço cloud
- Notebooks 01-03: Claude Vision API | Notebook 04: Qwen2.5-VL-72B (local)
- Deduplicação espacial pós-classificação: agrupa por nome, clusteriza por proximidade (centros < 100px), mantém menor bbox por cluster
- Output: lista JSON de componentes com nome, classe genérica, bounding box e provedor

*Passo 2c — Análise de topologia:*
- Enviar imagem completa + lista de componentes para análise topológica
- Notebooks 01-03: Claude Vision API | Notebook 04: Qwen2.5-VL-72B (local)
- Identificar: grupos/containers (VPC, Subnets, AZs), conexões entre componentes, fluxo de dados end-to-end
- Output: JSON com groups, connections e data_flow

**Passo 3 — Análise STRIDE (com contexto topológico):**
- Para cada componente detectado, gerar análise STRIDE completa
- Notebooks 01-03: Claude API | Notebook 04: Qwen2.5-VL-72B (local)
- Cada componente recebe análise nas 6 categorias: Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege
- Incluir nível de risco (Alto/Médio/Baixo) e contramedidas específicas do provedor
- Contexto topológico: em qual grupo está, com quem se conecta, posição no fluxo de dados
- Referenciar boas práticas (AWS Well-Architected / Azure Security Benchmark / Google Cloud Architecture Framework)
- Parser JSON robusto com 3 estratégias de fallback (clean+parse, raw extract, regex per-category)
- Responder em português do Brasil

**Passo 4 — Gerar relatório:**
- JSON estruturado (para integração) — inclui topologia e análise STRIDE
- HTML formatado (para apresentação) com CSS inline, sem dependências externas:
  - Resumo executivo com badges de risco coloridas (Alto=vermelho, Médio=amarelo, Baixo=verde)
  - Imagens da arquitetura embutidas em base64 (original + anotada com bboxes numerados)
  - Seção de topologia: fluxo de dados visual, grupos/containers, tabela de conexões
  - Tabela de componentes com numeração (#N), classe, localização topológica e provedor
  - Nota explicativa quando há componentes com mesmo nome em diferentes zonas
  - Análise STRIDE detalhada por componente com badge de localização, chips de conexões e categorias com nível de risco
  - Pode ser salvo como PDF pelo navegador (Ctrl+P)

## Arquiteturas de Teste

O sistema será testado com imagens disponíveis no repositório em "imagens/arquitetura *.png". Os notebooks 03 e 04 detectam automaticamente as imagens disponíveis na pasta do Google Drive.

**Arquitetura 1 — AWS:**
- Camada externa: Usuários SEI → AWS Shield → Amazon CloudFront → AWS WAF
- AWS Cloud > Region São Paulo > VPC com 3 Availability Zones (A, B, C)
- Cada AZ: Public Subnet (ALB) + Private Subnet (Auto Scaling SEI/SIP) + Private Subnet (storage)
- AZ A: EFS Multi-AZ + RDS Primary
- AZ B: RDS Secondary + ElastiCache Multiaz
- AZ C: Solr com Auto Scaling
- Serviços auxiliares: CloudTrail, KMS, Backup, CloudWatch, SES

**Arquitetura 2 — Azure:**
- Atores: Usuário (HTTP) e Developer
- Microsoft Entra (autenticação)
- API Management (Resource Group): API Gateway, Developer Portal, Logic Apps
- Backend Systems: Azure Services, SaaS Services, Web Services (REST, SOAP)
- Fluxo: 1 (User HTTP) → 2 (Entra Auth) → 3 (API Gateway) → 4 (Logic Apps) → 5 (Backend)

**Arquitetura 3 — GCP:**
- Clientes externos: Android, Chrome, iOS, Other
- Cloud Load Balancing (entrada)
- ESP Container (Extensible Service Proxy)
- API Container (backend)
- gcloud → Service Management → ESP
- Service Control → Cloud Console
- Fluxo: Clients → Cloud Load Balancing → ESP Container → API Container

## Estrutura de Notebooks

O projeto usa 4 notebooks no Google Colab:

**01_component_detection.ipynb** (GPU T4/A100, ~4 GB VRAM):
1. Montar Google Drive e carregar imagem
2. Passo 1: Claude Vision identifica provedor (AWS/Azure/GCP)
3. Passo 2a: Florence-2-large-ft detecta componentes (5 estratégias + tiling 3x3 com 20% overlap)
4. Passo 2b: Claude Vision classifica cada crop com nome exato do serviço
5. Passo 2c: Claude Vision analisa topologia (grupos, conexões, fluxo de dados)
6. Salvar JSON com componentes detectados e topologia
7. Visualizar bboxes numerados (#N) com supervision
Output: outputs/detections/componentes_arquitetura_X.json + annotated_arquitetura_X.png

**02_stride_analysis.ipynb** (CPU suficiente):
1. Carregar JSON de componentes (inclui topologia)
2. Montar prompt STRIDE contextualizado por componente (com posição topológica, conexões, fluxo)
3. Claude API gera análise STRIDE completa (em português do Brasil)
4. Gerar contramedidas específicas
5. Consolidar JSON
6. Gerar HTML com imagens base64, topologia, localização e conexões por componente
Output: outputs/reports/stride_arquitetura_X.json e .html

**03_consolidado.ipynb** (GPU T4/A100, ~4 GB VRAM):
1. Pipeline end-to-end completo (todos os passos 1-4) usando Claude API
2. Detecta automaticamente as imagens disponíveis na pasta do Google Drive
3. Visualização: imagem com bboxes numerados (#N) + resumo do relatório
4. Métricas: tempo, componentes detectados, ameaças identificadas
5. Estimativa de custo (Claude API + Colab GPU)
Output: Relatórios completos (JSON + HTML) + imagens anotadas

**04_local_pipeline.ipynb** (GPU A100 80 GB, ~42 GB VRAM):
1. Mesmo pipeline end-to-end do nb03, substituindo Claude API por Qwen2.5-VL-72B-Instruct-AWQ (local)
2. Gestão de memória GPU: carrega/descarrega Florence-2 e Qwen conforme necessário
3. Detecta automaticamente as imagens disponíveis na pasta do Google Drive
4. Visualização, métricas e comparativo de custo (API vs local)
5. Custo de API = US$ 0,00. Fallback: Qwen2.5-VL-7B-Instruct se 72B não estiver disponível
Output: Relatórios completos (JSON + HTML) + imagens anotadas
